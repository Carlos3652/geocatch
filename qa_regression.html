Findings — High Priority

HIGH HIGH-01 Streak multiplier applied BEFORE increment — first catch at streak=3 does NOT trigger 1.5x

Component: Catch handler, lines 637–646

Description: The streak multiplier is read and applied to the score before catch_streak is incremented. The upgrade thresholds (>=3, >=6) are then evaluated on the post-increment value and update streak_multiplier — but too late to affect the catch that just triggered the threshold. The player's 3rd catch scores at 1.0x; only the 4th catch scores at 1.5x. The player's 6th catch scores at 1.5x; only the 7th catch scores at 2.0x. The spec says 1.5x activates at 3 catches, implying the third catch should benefit.

Reproduction: 1. Start game. 2. Catch 3 creatures without hitting a bomb. 3. Observe float text on the 3rd catch — it still shows the 1x value. 4. Float text on the 4th catch shows 1.5x.

# Line 637-646 (current — BUG)
base_pts = caught["points"]
pts = int(base_pts * streak_multiplier)  # multiplier OLD value used
score += pts
inventory.append(caught["name"])
catch_streak += 1                        # increment AFTER scoring
if catch_streak >= 6:
    streak_multiplier = 2.0
elif catch_streak >= 3:
    streak_multiplier = 1.5              # updated too late

# Recommended fix: increment streak, update multiplier FIRST, then score
catch_streak += 1
if catch_streak >= 6:
    streak_multiplier = 2.0
elif catch_streak >= 3:
    streak_multiplier = 1.5
pts = int(base_pts * streak_multiplier)
score += pts

Impact: Player is systematically under-rewarded at every threshold boundary. Over a full game this can suppress scores by 50–100 points, affecting leaderboard placement. Also undermines the designed progression feel.

HIGH-02 HIGH-02 Game-over name-entry inaccessible when score=0 — player stuck with no escape

Component: Game-over event handler, line 787; game-over input handler, lines 672–681

Description: When the game ends with score=0, is_new_high_score is forced to False (line 787: score > 0 and ...). However, the name-entry UI and submit logic (score_saved path) are still rendered because they are gated on score_saved, not is_new_high_score. The player sees the name-entry pill and can type a name, but pressing ENTER has no effect because the save branch silently calls save_high_score("NAME", 0) — which IS reachable — then sets score_saved = True. This means a score of 0 can pollute the high-score file. Separately, the only way to leave the game-over screen is pressing R after score_saved is True. If the player presses R without saving first (skipping name entry), there is no handler for that — pressing R when score_saved=False does nothing, and there is no ESC/Q handler in game_over state. The player must enter a name and submit before they can leave.

Reproduction: 1. Start game. 2. Immediately walk into all bombs until time expires with 0 score (or just wait 60s). 3. On game-over screen press R — nothing happens. 4. No instruction is shown to tell the player how to proceed without entering a name.

# Line 787
is_new_high_score = score > 0 and (...)   # 0-score game: False

# Lines 672–673 — R key only works AFTER score_saved
if event.key == pygame.K_r and score_saved:
    game_state = "character_select"

# No handler for: R when score_saved=False, ESC, Q in game_over state
# Also: save_high_score(name, 0) is never blocked — 0 scores enter leaderboard

Recommended fix: (a) Allow R key to return to menu even when score_saved is False (skip name entry for zero or low scores if desired). (b) Add an ESC or Q handler in game_over state. (c) Guard save_high_score so scores of 0 are not written. (d) Display a "Press R to skip / Return to Menu" hint unconditionally.

Impact: High — a player who finishes with 0 points is trapped on the game-over screen until they submit a name, with no visible instruction explaining this.

Findings — Medium Priority

MED MED-01 Proximity label and catch pop animation positions not offset by screen shake

Component: Draw loop, lines 1118–1133, 1101–1109

Description: During screen shake all world elements (rocks, bombs, creatures, trainer) are correctly offset by (_shake_ox, _shake_oy). However, the proximity label pill is drawn using the unshaken creature world coordinates (c["x"], c["y"]) as anchor, with no _shake_ox / _shake_oy added. This causes the label to appear misaligned from its creature during the 6-frame shake window. The catch pop animations at lines 1108–1109 do apply shake offsets — but the proximity label does not.

# Line 1123 — shake offset MISSING on lbl_x anchor
lbl_x = int(closest_c["x"]) - lw // 2 - pad_x   # should be int(closest_c["x"]) + _shake_ox - ...
lbl_y = max(4, int(closest_c["y"]) + closest_c.get("_bob", 0) - 50 - lh - pad_y * 2)
#                                                   ^^ _shake_oy missing too

Impact: Visual inconsistency — label detaches from creature for ~6 frames on each bomb hit. Minor but noticeable. Also confirms the HUD pills (score, timer, streak) are correctly unshaken since they use fixed pixel coords.

MED MED-02 Best Scores popup allocates a new SRCALPHA Surface every frame it is visible

Component: Character select draw block, lines 923–937

Description: The "How to Play" popup is correctly pre-rendered at startup into _cs_popup_how. The "Best Scores" popup, however, allocates a fresh pygame.Surface((_CS_POPUP_W, _CS_POPUP_H), pygame.SRCALPHA) every draw frame, fills it, draws border and text, then blits to screen. At 60 FPS this is 60 surface allocations per second while the popup is open, producing GC pressure. This was flagged in the QA Pass 3 notes as MED-03 and remains unresolved.

# Line 924 — executes every draw frame while _cs_popup == "scores"
_sp = pygame.Surface((_CS_POPUP_W, _CS_POPUP_H), pygame.SRCALPHA)   # NEW alloc
_sp.fill((26, 26, 46, 240))
pygame.draw.rect(_sp, ...)
# ... blit all score rows ...

Recommended fix: Pre-render into a module-level _cs_popup_scores surface; rebuild it only when high_scores changes (on save). Follow the same pattern as _cs_popup_how.

Impact: Performance regression; observable on lower-end hardware or after the popup has been open for many seconds. Also inconsistent with the established caching pattern.

MED MED-03 Score and Timer HUD pills call font.render() every frame — two per-frame allocations

Component: Playing draw loop, lines 1165, 1183

Description: font.render(f"Score: {score}", ...) and font.render(f"Time: {time_left}s", ...) are called on every draw frame. While the score changes only on catches/bomb-hits and time_left changes at most once per second, these calls happen 60 times per second unconditionally. Each call allocates a new pygame Surface. The streak text at line 1170 has the same issue: small_font.render(...) per frame. Compare against _go_cache which correctly pre-renders on game-over.

# Line 1165 — every draw frame
screen.blit(font.render(f"Score: {score}", True, SCORE_GOLD), ...)

# Line 1183 — every draw frame
screen.blit(font.render(f"Time: {time_left}s", True, ...), ...)

Recommended fix: Cache the last rendered score and time_left surfaces; re-render only when the values change. Pattern: if score != _cached_score: _score_surf = font.render(...); _cached_score = score.

Impact: 2–3 extra allocations per frame during gameplay. At 60 FPS: ~180 allocations/sec. Contributes to garbage collection pauses on low-end hardware.

MED MED-04 Bubble surfaces allocated per-frame on game-over screen — 14 × 60 = 840 allocs/sec

Component: Game-over draw loop, lines 948–954

Description: For each of the 14 bubbles, a 50×50 SRCALPHA surface is created, filled, and drawn every frame. This is the most expensive per-frame allocation in the codebase. The bubble position changes each frame (y moves up) but the surface size and color are fixed per bubble lifecycle.

# Line 950 — runs 14 times per frame, 60 frames/sec = 840 new Surfaces/sec
_bs = pygame.Surface((_bubble_surf_size, _bubble_surf_size), pygame.SRCALPHA)
pygame.draw.ellipse(_bs, (*b["color"], 90), ...)

Recommended fix: Pre-render one surface per bubble color/size combination on game-over entry (when _go_bubbles is built), store as b["surf"], and blit with position-only updates each frame. The pre-rendered _go_flash_surf at line 271 demonstrates the intended pattern.

Impact: Highest allocation rate in the file. Likely to cause frame drops on integrated graphics during the game-over celebration, precisely when visual fidelity matters most.

MED MED-05 Pause state: Q quits without resetting game state — resuming later shows stale data

Component: Pause event handler, line 667–668

Description: When the player presses Q in the pause menu, game_state = "character_select" is set directly without calling reset_game(). The globals score, catch_streak, streak_multiplier, _escalation_triggered, creatures, bombs, _total_paused_ms, and _pause_start_ms all retain their in-game values. If the player returns to the main menu, sees the select screen, and immediately starts again (pressing ENTER or 1-4), reset_game() IS called from the character-select event handler, so the next game is clean. However, if any code path re-enters "playing" without going through character-select (currently not possible, but relevant for future additions), stale state would leak. More concretely: _pause_start_ms is NOT added to _total_paused_ms before quitting. If telemetry or stats were ever read at this point, the timing would be wrong. Additionally, the per-trainer best score is NOT saved on Q-quit (only saved when time_left <= 0), so an in-progress personal best is discarded silently.

# Lines 667-668
elif event.key == pygame.K_q:
    game_state = "character_select"   # no reset_game(), no stats save

Impact: Stats not saved on Q-quit; in-progress personal best lost. Low corruption risk currently, moderate risk as a future regression vector.

LOW LOW-04 Escalation adds 5th bomb immediately on trigger frame — mid-wave bomb appears without any warning

Component: Late-game escalation, lines 776–779

Description: When time_left first drops below 15, the 5th bomb is appended via bombs.append(_spawn_bomb_random()) with no visual fanfare, audio cue beyond the existing tick sound, or brief placement delay. The bomb materialises at a random position that could be on top of the player (the 80px minimum distance is from the player position at spawn time, but the player could have moved into that zone by the next frame). No cooldown is set at escalation time, so the player can be hit on the very first frame the bomb exists if they were unlucky about position.

Recommended fix: Either play a distinct escalation sound, show a brief "DANGER ZONE!" flash text, or impose a short 30-frame cooldown on the new bomb at spawn. Also add the new bomb to the existing cooldown window so it inherits safety time.
