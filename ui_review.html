Start Screen — Character Select
Lines 316–584
▼

Current State Audit

The character select screen stacks five distinct information zones in a 700px-tall window: a full-world background, a dark overlay, a 760×320 trainer panel, four trainer cards, a CTA button, and two side-by-side mini panels (How to Play + Top Scores) — all rendered simultaneously. Every zone uses its own background fill, border, and font size, with no clear visual hierarchy. The result is dense and competing, especially on the cramped bottom strip.

  • Two full info panels (How to Play + Top Scores) are visible alongside the selection UI at all times. Both fight for attention with the primary task (picking a trainer). lines 567–583
  • The How to Play panel lists six individual bullet lines in 14px tiny_font, including per-creature point values. This is a manual/documentation panel, not a start screen element. lines 357–366
  • The CTA button reads "PRESS ENTER OR [1-4] TO START" — this is a 33-character instruction disguised as a button. Primary CTAs should say what happens, not how to trigger it. line 376
  • The title "GEOCATCH" and subtitle "OVERLAND PARK EDITION" sit directly above "— CHOOSE YOUR TRAINER —" with only 10px of breathing room between all three. No visual separation makes this read as one dense block. lines 515–523
  • Trainer cards show three stacked labels under the sprite: "Trainer 1", "[1]", and a "SELECTED" badge — three labels for one card. The keyboard hint [1] duplicates information already on the CTA button. lines 549–558
  • The bottom mini panels (_CS_MINI_H = HEIGHT - _CS_MINI_Y - 10) only have ~78px of height, causing bullet text to be cut off or very tightly packed. lines 321–322
  • Six bullet points cover controls, catch mechanics, and all five creature point values. This front-loads cognitive load before the player has even started once. lines 358–366
Design Variants — A/B/C/D
A
Focused Stage
Recommended

Remove all secondary info panels from the start screen entirely. The screen has one job: pick a trainer and start. Instructions move to a first-run tooltip or pause menu. Top scores become a scrolling ticker or appear only after a game completes. The trainer cards expand to fill the space, becoming the visual hero.

Pros
  • Single clear hierarchy
  • Cards get 40% more room
  • Zero cognitive overload
  • CTA always above the fold
Cons
  • Controls not visible upfront
  • Scores need new location
  • New players need discovery

Best for: Returning players, family use, quick restarts

LOW EFFORT — 45 min
B
Tab Drawer
Progressive Disclosure

Keep info panels but hide them behind a "?" icon and "Scores" link at the top-right. Clicking either expands a slim drawer over the bottom 120px of the screen. Primary UI stays clean; secondary content accessible but not intrusive.

Pros
  • Info stays available
  • Clean default state
  • Good for new players
Cons
  • More code to manage
  • Drawer state persists
  • Medium Pygame effort

Best for: Balancing discovery and cleanliness

MED EFFORT — 2 hrs
C
Card Spotlight
Minimal

Keep only one panel — the trainer cards. Collapse How-to and Scores into two small icon-text links ("How to Play" and "Best Scores") below the CTA button. Selected trainer gets a larger card with a brief name flavor text line. Top scores appear only in the game over screen where they are contextually relevant.

Pros
  • Scores in context (game over)
  • Minimal draw calls
  • Icon links are small
Cons
  • Icon links need hover
  • No up-front score view

Best for: Streamlined, fast-start experience

LOW EFFORT — 1 hr
D
Score Ticker
Ambient Info

Remove both mini panels. Replace Top Scores with a slim horizontal ticker strip at the very bottom of the screen (18px tall, scrolling "DAD — 2270 pts | LEO — 475 pts | ..."). How to Play condenses to 3 icon-glyphs with single-word labels (MOVE / CATCH / DODGE) centered below the CTA button. Everything fits in one vertical scan.

Pros
  • Scores always visible
  • Controls fit in 3 words
  • Clean centre layout
  • Family score bragging
Cons
  • Scrolling ticker adds code
  • Small text at 18px
  • Ticker may distract

Best for: Family / competitive household context

End Screen — Game Over
Lines 585–626
▼

Current State Audit

The game over screen is a flat black fill with text items stacked vertically at fixed y offsets from a single go_y = 100 anchor. There is no card, no panel, no background hierarchy — just text on black. The result looks unfinished and fails to give the score the visual weight it deserves. The name entry flow and high score table appear at the same visual level as everything else.

  • "TIME'S UP - GAME OVER" is the headline but renders at the same font size (36pt) as "Final Score: 340" and "Creatures Caught: 5". No typographic hierarchy — all three read as equals. lines 499–502
  • screen.fill(BLACK) — pure black background is the harshest possible choice. No panel, no gradient, no atmospheric treatment. The whole screen is dead space except for 7 text items centered down the middle. line 587
  • The name entry prompt is a plain text string ("Enter name (5 chars): _") with no visual input field treatment — no box, no border, no focus indicator. lines 616–618
  • The creature color dots row (radius 10, spacing 28) appears at go_y + 148 but is only rendered if caught_count > 0. When score is 0, nothing appears here — the layout visually collapses with no placeholder. lines 601–610
  • High score table entries use plain white text for all rows except rank 1 (gold). Ranks 2–5 are visually indistinct. No row separation, no alternating shade. lines 622–626
  • "Press R to return to menu" — R is a non-obvious key. A visible button or [ENTER] would be more consistent with the start screen's own CTA convention. line 614
  • New high score state renders a glow/gold version of the same string at -36px offset — essentially a shadow trick. At 24pt font, this is subtle enough to be easily missed. Needs a larger visual treatment to feel celebratory. lines 591–593
Design Variants — A/B/C
A
Score Card
Recommended

Replace the flat black screen with a centered card panel (680x420px, rounded 16px, PANEL_BG fill with accent border). Inside the card: giant score number at 64pt, headline at 28pt, creature dots, name entry as a styled input box, and the high score table — all confined to one structured container. Background gets a subtle radial gradient (dark center, slightly lighter edges) using two blended surface layers.

Pros
  • Score feels earned + weighted
  • Clear spatial hierarchy
  • Input box looks interactive
  • Consistent with menu panels
Cons
  • Needs panel surface alloc
  • May clip on small monitors

Best for: Making the score feel like a reward, not a report

Atmospheric

Pros

  • Avoids dead black
  • Reuses world surf asset
Cons
  • Map distracts from score
  • High contrast needed

HUD — Gameplay Overlay
Lines 629–743
▼

Current State Audit

The gameplay HUD is actually the least noisy of the three screens — just two pills (Score left, Timer right) and the proximity label. However, the catch radius circle, the pulse ring, and the proximity label all overlap the player character in the center of the screen. During active gameplay, the player is surrounded by three overlapping indicators at once.

  • The catch radius is a plain 1px gray circle (r=55) around the player at all times. It has no fill, no opacity treatment, and renders as a clinical laboratory indicator in the middle of a colorful game world. At r=55 it is quite large and always visible. line 701
  • When a creature is within catch range, three overlapping elements appear simultaneously around the player: (1) the gray catch ring, (2) the orange pulse ring, and (3) the proximity label pill floating above the creature. This is three concurrent visual events. lines 673–710
  • The proximity label shows: "Fire Drake — 50 pts | SPACE to catch" — this is 30+ characters of text at 20pt. The "SPACE to catch" portion can be shown once (as a tip) and then hidden once the player has caught a creature. Showing it every single catch interaction is repetitive for experienced players. line 680
  • Score pill reads "Score: 340" and Timer pill reads "Time: 45s" — the labels "Score:" and "Time:" consume visual space that the numbers alone would communicate. Compare to minimalist HUDs where "340" + a trophy icon is instantly readable. lines 733, 742
  • The two pills (210px wide Score, 180px wide Timer) use different widths and sit at different y positions visually due to font metrics variation. They should be the same height and same width for balance. Current: score pill 210px, timer pill 180px. lines 730, 739
  • The bomb flash is a full-screen SRCALPHA surface allocated every frame while active (3 frames). This should be pre-allocated like other effect surfaces. lines 717–720

Sprite & Character Visual Polish
Lines 656–714
▼

Why Sprites Look Blocky — Root Cause Analysis

The sprites themselves are loaded with smoothscale (line 37 and 99), which is correct. The "blocky" appearance is not a scaling artifact — it is a contextual integration problem. The sprites are PNG rectangles placed directly onto a flat grass background with no visual bridging. Several factors contribute:

  • No sprite outline / edge separation from background. On grass-colored backgrounds, green-tinted creature edges and trainer edges blend directly into the world color. There is nothing separating the sprite silhouette from the world. This makes them appear as "pasted-on" rectangles. lines 666–668 (creatures), 712–713 (trainer)
  • Type-coded base circles are small (56x18 ellipses) and positioned too low. They appear as a colored stripe at the sprite's feet, not as a visible "element type" indicator. At 56x18px for a 60x60 sprite, the circle is barely a sliver. lines 135–137, 660
  • Trainer sprite has no grounding treatment beyond the shadow ellipse. The shadow (_shad_trainer: 60x16px, alpha 70) is drawn, but the trainer PNG is placed directly on the world with no backing circle or outline stroke. Against road-colored pixels (210,210,210) the trainer's lighter pixels become invisible. lines 698, 712–713
  • Pulse ring is centered incorrectly relative to the trainer sprite. The ring draws at (player_x - 62, player_y - 10 - 62) — it centers on the sprite's approximate torso, not feet. This creates an aura that floats above the trainer rather than surrounding them, which looks disconnected. line 710
  • Bob animation amplitude is 5px on a 60x60 sprite. At 5px this is subtle. For family/kids audience, 7-8px makes the creature feel more alive and expressive. Shadow ellipse does not bob in sync with the creature. line 657
  • No idle animation for the trainer/player character. The player sprite is completely static during movement and idle states. Even a 2px vertical bob would add life. This is especially noticeable for kids. lines 712–713
Visual Treatment Options — 1 through 5
1
Soft Edge Ring
Highest Impact / Lowest Effort

Draw a thin 2px white or cream outline circle just behind each sprite (drawn before the sprite blit). This creates a sticker-style edge that separates the sprite from the background regardless of what color terrain is underneath. Works for both trainers and creatures. No alpha blending needed.

Pros

  • Eliminates "blending into grass"

NOTE
Shadow scaling performance note: pygame.transform.scale() allocates a new Surface every call. For 8 creatures at 60fps this is 480 allocations/second — acceptable but not ideal. If profiling shows it as a hotspot, pre-allocate 5 shadow sizes (100%, 92%, 85%, 78%, 72%) at startup and select the nearest size by index based on abs(bob). This reduces allocs to zero at the cost of 5 static surfaces (5 × ~50×12px = negligible).
ORDER
Critical draw order for full effect (per entity):
1. Glow aura (SRCALPHA, behind everything)
2. Ground platform ellipse (type color, opaque)
3. Backing circle (cream, opaque)
4. Outline ring (2px, gray/white)
5. Scaled shadow ellipse
6. Sprite blit
Do NOT draw the shadow after the backing circle — it will appear in front of it.
