Critical CRIT-01 Per-frame score_big_font.render() during 2-second count-up animation line 1147

During the score count-up animation (_go_anim_done == False), the code falls through to the else branch and calls: 

_score_text = score_big_font.render(str(_go_anim_score), True, SCORE_GOLD)

score_big_font is a 72pt bold Arial surface. Every call to font.render() allocates a new SDL surface and performs full glyph rasterisation. At 60 fps over 2 seconds this produces 120 large surface allocations. On a score of 1000 the speed floor is max(200, 500) = 500 pts/s, meaning each frame has a different integer value so no frame-to-frame reuse is possible. This is the single highest-cost per-frame allocation in the game-over state and causes visible stutter on entry and throughout the animation on integrated graphics.

The existing "_score_final" cache at line 1142 only activates after the animation completes, leaving the animation frames completely unoptimised.

Fix: Use a dict cache keyed on the displayed integer value. At most ~120 entries across 2 seconds, all cleared on new game. 

# At game-over init (inside the time_left <= 0 block, ~line 893):
_go_cache["_score_surfs"] = {}   # int -> Surface, cleared each run

# In draw block, replace lines 1142-1148:
if _go_anim_score not in _go_cache["_score_surfs"]:
    _go_cache["_score_surfs"][_go_anim_score] = score_big_font.render(
        str(_go_anim_score), True, SCORE_GOLD)
_score_text = _go_cache["_score_surfs"][_go_anim_score]

Pre-seed the cache with the final score at init time so the most common case (animation already complete) is a guaranteed hit with no render call.

Critical CRIT-02 set_alpha() silently ignored on _go_flash_surf — high-score flash never fades lines 314–315, 1073–1074

_go_flash_surf is created with pygame.SRCALPHA: 

_go_flash_surf = pygame.Surface((WIDTH, HEIGHT), pygame.SRCALPHA)
_go_flash_surf.fill((255, 140, 0, 80))

During game-over draw, the code attempts to fade it out: 

_go_flash_surf.set_alpha(int((_go_flash_timer / 0.5) * 80))
screen.blit(_go_flash_surf, (0, 0))

This is a documented Pygame gotcha. Surface.set_alpha() is silently ignored on any surface created with pygame.SRCALPHA. Per-pixel alpha and surface-level alpha are mutually exclusive in Pygame's SDL backend. The call returns without error but has zero effect. The flash renders at a constant alpha of 80 for the entire 0.5 s lifetime of _go_flash_timer, then disappears abruptly. The intended fade-out animation does not function at all.

Fix — Option A (simplest): Change _go_flash_surf to a non-SRCALPHA surface and use set_alpha() for whole-surface opacity: 

# Allocation (line 314):
_go_flash_surf = pygame.Surface((WIDTH, HEIGHT))
_go_flash_surf.fill((255, 140, 0))

# Draw block (lines 1072-1074):
if _go_flash_timer > 0:
    _go_flash_surf.set_alpha(int((_go_flash_timer / 0.5) * 80))
    screen.blit(_go_flash_surf, (0, 0))   # now actually fades

Fix — Option B (keep SRCALPHA, refill per frame): 

if _go_flash_timer > 0:
    _alpha = int((_go_flash_timer / 0.5) * 80)
    _go_flash_surf.fill((255, 140, 0, _alpha))
    screen.blit(_go_flash_surf, (0, 0))

Option A is marginally faster; Option B requires fewer allocation changes.

lw, lh = label_text.get_width(), label_text.get_height()

High HIGH-01 Firework max_life initialised to 0.0 before assignment — division-by-zero risk lines 919, 925–929, 1078

Each firework particle is constructed in two steps: 

_go_fireworks.append({
    ...
    "life": random.uniform(1.5, 3.0),
    "max_life": 0.0,                          # <-- placeholder
    ...
})
_go_fireworks[-1]["max_life"] = _go_fireworks[-1]["life"]  # <-- then patched

The two-step assign-then-patch pattern leaves a window where max_life == 0.0. In the current single-threaded loop the patch always completes before any draw call, so it does not crash in practice. However the draw code at line 1078 computes: 

_fw_alpha = max(0, min(255, int(255 * fw["life"] / fw["max_life"])))

A ZeroDivisionError would fire if any code path ever iterated _go_fireworks between the append and the patch (e.g. a future refactor that moves population to a helper function, or adding a background thread). The pattern is also simply harder to read than a single-expression constructor.

Fix: Compute max_life once and use it in the constructor dict directly: 

for _ in range(30):
    _ml = random.uniform(1.5, 3.0)
    _go_fireworks.append({
        "x":        random.randint(50, WIDTH - 50),
        "y":        random.randint(30, HEIGHT // 2),
        "vx":       random.uniform(-60, 60),
        "vy":       random.uniform(-80, 20),
        "life":     _ml,
        "max_life": _ml,    # never 0.0
        "color":    random.choice(list(CREATURE_COLORS.values())),
        "r":        random.randint(2, 5),
    })
High HIGH-02 Firework alpha computed but never applied — particles don't fade out lines 1078–1082

The draw loop correctly computes per-particle alpha based on remaining lifetime: 

_fw_alpha = max(0, min(255, int(255 * fw["life"] / fw["max_life"])))

But the circle draw call on line 1079 ignores it entirely: 

pygame.draw.circle(screen, (*fw["color"], ), (int(fw["x"]), int(fw["y"])), fw["r"])

(*fw["color"],) unpacks a 3-tuple RGB. pygame.draw.circle() on a non-SRCALPHA target surface ignores any 4th alpha channel in the color tuple. The variable _fw_alpha is computed, used only for the trail-dot branch decision (if _fw_alpha < 200), and then discarded. Firework particles therefore appear at full brightness and vanish instantly at life <= 0 instead of fading. On a high-score screen this looks like a bug — particles pop out of existence abruptly rather than dissolving.

Fix — simplest (no new surfaces): Modulate the RGB color by the life fraction directly: 

for fw in _go_fireworks:
    _frac = fw["life"] / fw["max_life"]
    _col = tuple(int(c * _frac) for c in fw["color"])
    pygame.draw.circle(screen, _col, (int(fw["x"]), int(fw["y"])), fw["r"])
    if _frac < 0.78:  # show trail when fading
        _trail_col = tuple(int(c * _frac * 0.5) for c in fw["color"])
        pygame.draw.circle(screen, _trail_col,
            (int(fw["x"] - fw["vx"] * 0.02), int(fw["y"] - fw["vy"] * 0.02)),
            max(1, fw["r"] - 1))

This avoids any surface allocation and produces a smooth colour-based fade that reads as natural particle decay.

High HIGH-03 Fader alpha formula duplicated between catch guard and draw path — maintenance divergence risk lines 709–713 vs 1273–1277

The effective alpha of a Shadow Phantom is computed in two separate places with copy-pasted logic. The catch guard (SPACE handler): 

_eff_alpha = c.get("spawn_alpha", 1.0) * 255
if c.get("behavior") == "fader":
    _fd = math.hypot(c["x"] - player_x, c["y"] - player_y)
    _eff_alpha = min(_eff_alpha, max(50, 255 - (_fd - 100) * 2.05))
if _eff_alpha < 128:
    continue

The draw path (creature render):

_c_alpha = int(c.get("spawn_alpha", 1.0) * 255)
if c.get("behavior") == "fader":
    dist = math.hypot(c["x"] - player_x, c["y"] - player_y)
    fader_alpha = int(max(50, min(255, 255 - (dist - 100) * 2.05)))
    _c_alpha = min(_c_alpha, fader_alpha)

The draw path correctly adds min(255, ...) inside the fader_alpha computation; the catch guard omits it. Both happen to produce the same result at all reachable distances because the outer min(_eff_alpha, ...) in the catch guard clamps the upper bound anyway. However the formulas are semantically inconsistent and will silently diverge the moment either is modified. Any change to catchability thresholds (e.g. making phantoms harder to catch) must be applied in two places.

Fix: Extract a single helper function used by both: 

def _effective_alpha(c: dict, px: float, py: float) -> int:
    """Returns the effective display/catch alpha [0..255] for a creature."""
    alpha = c.get("spawn_alpha", 1.0) * 255
    if c.get("behavior") == "fader":
        dist = math.hypot(c["x"] - px, c["y"] - py)
        fader_alpha = max(50.0, min(255.0, 255.0 - (dist - 100) * 2.05))
        alpha = min(alpha, fader_alpha)
    return int(alpha)

# Catch guard:
if _effective_alpha(c, player_x, player_y) < 128:
    continue

# Draw path:
_c_alpha = _effective_alpha(c, player_x, player_y)

Calling the helper twice per creature per frame (once in the event handler, once in draw) is ~10 ns each and completely negligible.

High HIGH-04 Drifter axis-split collision allows creature to enter lake after X-axis move is accepted lines 835–846

The drifter collision check splits X and Y axes independently: 

if _nx < 80 or _nx > 800 or in_lake(_nx, c["y"]) or ...:
    c["vx"] = -c.get("vx", 0)
else:
    c["x"] = _nx          # <-- c["x"] updated here
if _ny < 80 or _ny > HEIGHT - 80 or in_lake(c["x"], _ny) or ...:
    # c["x"] above is now the POST-MOVE value, not pre-move
    c["vy"] = -c.get("vy", 0)
else:
    c["y"] = _ny

After the X check accepts the move and writes c["x"] = _nx, the Y check uses the updated c["x"]. This means the Y check is testing the lake boundary at the new X position, not the old one. For diagonal movement near the edge of a lake ellipse, it is possible for the X check to pass (new_x is outside the lake at old_y), then the Y check to also pass (new_x is outside the lake at new_y), but the combined position (new_x, new_y) to sit inside the lake ellipse. A drifter moving SE at ~45 degrees into the lake at Millrace Pond can thread through the axis-split check and become lodged inside the water.

Fix: Perform the collision test on the proposed combined position before committing either axis: 

elif beh == "drifter":
    _nx = c["x"] + c.get("vx", 0) * dt
    _ny = c["y"] + c.get("vy", 0) * dt
    _ox, _oy = c["x"], c["y"]   # save originals

    # Test X axis with original Y
    _ok_x = (80 <= _nx <= 800
              and not in_lake(_nx, _oy)
              and not any(math.hypot(_nx - rx, _oy - ry) < 55 for rx, ry in rocks))
    # Test Y axis with original X
    _ok_y = (_ny >= 80 and _ny <= HEIGHT - 80
              and not in_lake(_ox, _ny)
              and not any(math.hypot(_ox - rx, _ny - ry) < 55 for rx, ry in rocks))

    if _ok_x:
        c["x"] = _nx
    else:
        c["vx"] = -c.get("vx", 0)
    if _ok_y:
        c["y"] = _ny
    else:
        c["vy"] = -c.get("vy", 0)

Using the original axis values for each independent test prevents the compound-position lake penetration.

Medium MED-02 Game-over globals not cleared when R/Q exits to character select without calling reset_game() lines 754–757

When the player presses R or Q from game-over: 

if event.key == pygame.K_r:
    game_state = "character_select"
elif event.key == pygame.K_q:
    game_state = "character_select"

reset_game() is not called at this point. The game-over globals _go_cache, _go_caught_keys, _go_fireworks, _go_anim_score, and _go_flash_timer all retain their values from the completed game. If the character select screen or any future code reads these variables before a new game starts, it will see stale data. reset_game() is called when the user clicks START or presses ENTER/1234 from character select, so the state is corrected before gameplay begins. But between the R/Q press and the next reset, the module namespace is in a partially-stale state.

Additionally, _go_fireworks continues to iterate in the game_over update block only — once state switches to character_select the particles are frozen in memory but not freed. Minor, but worth cleaning up.

Fix: Add an explicit cleanup on the R/Q transition: 

if event.key in (pygame.K_r, pygame.K_q):
    _go_cache = None
    _go_fireworks = []
    _go_caught_keys = set()
    game_state = "character_select"

Medium MED-04 Game-over layout overlaps on high-score path — NEW HIGH SCORE banner collides with showcase row lines 1094–1119

The layout is built top-down from go_y = 20:

  • Tier message at go_y = 20 (40px tier_font → ~48px tall)
  • NEW HIGH SCORE banner at go_y + 48 = 68 (24px bold → ~32px, bottom edge ~100px)
  • Showcase row always starts at go_y + 90 = 110 (hardcoded)

On the high-score path the banner occupies y=68 to y~=100. The showcase row starts at y=110 — leaving only 10px of separation between the banner bottom and the sticker tops. On systems where Arial renders slightly larger (common on high-DPI displays or when a fallback font is used), this gap disappears and the banner text renders on top of the stickers. On the non-high-score path the showcase has 90px below the tier message which is comfortable.

Fix: Derive _showcase_y from layout content rather than a hardcoded constant: 

_TIER_H = _go_cache["tier"].get_height() if _go_cache else 48
_HS_H   = (_go_cache["hs"].get_height() + 8) if (is_new_high_score and _go_cache and "hs" in _go_cache) else 0
_showcase_y = go_y + _TIER_H + _HS_H + 12   # 12px breathing room

All subsequent elements (_mid_y, _stats_y, _bot_y) already derive from _showcase_y, so the cascade is automatic once this anchor is corrected.

Medium MED-06 Pause screenshot captures the previous frame — active bomb flash can bleed into pause overlay lines 701–703

When the player presses ESC to pause: 

_pause_start_ms = pygame.time.get_ticks()
_pause_screenshot.blit(screen, (0, 0))   # captures backbuffer = previous frame
game_state = "paused"

screen at this point holds the backbuffer from the previous draw pass. The current frame has not been drawn yet (the draw block runs from line 1224 onward). This means: (a) the snapshot is one frame stale (invisible at 60 fps but technically incorrect), and (b) if the player walks into a bomb on the same frame they press ESC, the pause screenshot captures the red bomb flash overlay from the previous frame, making the pause screen look like an error state rather than a clean pause.

# Global:

_pause_start_ms = pygame.time.get_ticks()
game_state = "paused"

# At the END of the playing draw block (after all draws, before display.flip):
if _pending_pause:
    _pause_screenshot.blit(screen, (0, 0))
    _pending_pause = False
Medium MED-07 Text cursor blink phase not reset on keystroke — cursor flickers immediately after typing line 1203

The cursor blink is driven by wall-clock time: 

cursor = "" if len(name_input) >= 5 else ("|" if int(_ticks * 2) % 2 == 0 else " ")

_ticks is derived from pygame.time.get_ticks() and runs continuously. When the player types a character, the cursor phase does not reset — depending on where the clock lands, the cursor may show "|" on the character-typed frame and then flip to " " on the very next frame (16ms later). Standard text-entry UX resets the blink timer on each keystroke so the cursor is always visible immediately after the user types, providing clear feedback that the input was registered.

Fix: Track the last-keystroke time and compute blink relative to it: 

# Global:
_name_cursor_reset_t = 0.0

# In KEYDOWN handler for name_input (after any modification to name_input):
_name_cursor_reset_t = pygame.time.get_ticks() / 1000.0

# In draw, replace line 1203:
_cursor_age = _ticks - _name_cursor_reset_t
cursor = ("" if len(name_input) >= 5
          else ("|" if int(_cursor_age * 2) % 2 == 0 else " "))

Low LOW-02 _gen_tone() pure Python loop blocks startup for 120–320 ms lines 40–64

_gen_tone() iterates a Python for loop over n = int(44100 * duration_ms / 1000) samples. The gameover tone uses duration_ms=400 → 17,640 iterations. Four tones are generated synchronously at startup, totalling 120–320 ms of blocked startup time before the first frame is drawn on slow hardware (low-end laptops, integrated graphics, Raspberry Pi). The function also recomputes int(n * 0.75) and a full sine value per sample.

Fix options in order of impact: (1) Ship pre-generated WAV files; replace generation with pygame.mixer.Sound("sounds/gameover.wav"). (2) If runtime generation is required, use numpy to vectorise the loop (~100x faster, <3 ms per tone). (3) Generate tones lazily (on first play rather than at import) so startup is not blocked.
